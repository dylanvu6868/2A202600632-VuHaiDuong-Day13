## 4. Incident Response (Group)
- [SCENARIO_NAME]: `rag_slow`
- [SYMPTOMS_OBSERVED]: After `POST /incidents/rag_slow/enable`, the `/dashboard` Latency panel showed `latency_p95`/`latency_p99` jump from a ~150ms baseline to ~2500ms — right at the edge of the 3000ms SLO and inside the window the `high_latency_p95` alert (`> 3000ms for 10m`) is designed to catch one notch earlier. `/chat` end-to-end response times for every request rose to ~2.5s.
- [ROOT_CAUSE_PROVED_BY]: Log line for `correlation_id: req-d38b4a52` (`response_sent`, `latency_ms: 2500`) plus the corresponding Langfuse trace, whose single `run` generation span duration matched the same ~2.5s — pointing directly at `app/mock_rag.py:retrieve()`, which does `time.sleep(2.5)` when `STATE["rag_slow"]` is true (`app/incidents.py`). Notably, this request was also a response-cache hit (`cache_hit: true`, `cost_usd: 0.0`) — proving the cache-based cost optimization (Section 6) does **not** mask the latency incident, because `retrieve()` runs before the cache lookup.
- [FIX_ACTION]: `POST /incidents/rag_slow/disable` restored latency to baseline (~150ms). In a real system this maps to: investigate the vector-store/retrieval backend latency, add a timeout + fallback to `retrieve()`, and/or cache frequent queries.
- [PREVENTIVE_MEASURE]: `config/alert_rules.yaml` `high_latency_p95` threshold was tightened from 5000ms/30m to 3000ms/10m specifically so a single `rag_slow`-class incident (≈2500ms) is caught close to the SLO boundary instead of 2000ms past it. Longer-term: give `retrieve()` its own child span/metric so retrieval latency is visible independent of total agent latency.
